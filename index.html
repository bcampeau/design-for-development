Design at Alley Interactive

We have thought about the biggest challenges facing the design and development process. We started solely as a software development agency and after years of working with outside agencies, we felt the best way to solve the problem was to start our own. The lessons that we learned can help everyone here design more effectively.

Eliminating the Waterfall

(and the dreaded "Handoff")

The waterfall leads to the handoff. Designers think their job is done and now developers build it. In this model, there is often very little communication during the design phase. This leads to two main issues that are root of all problems I'll talk about today...

Incomplete Designs

Missing styles, missing states, missing features, lack of clear direction. The designs simply don't cover everything the developer needs to build the site.

Impractical Designs

Are You @$%#&! Kidding Me?

Unbuildable features lead to poorly written code that's trying to handle edge cases that don't really need to exist. It leaves developers asking - was this choice made for a reason?
We've avoided this by establishing six key pillars to our design practice, the first of which is...

Multidisciplined Teams Win

Conscious choice in 2013 to become a full service digital agency, uniting product strategy, design, user experience, and website development. Eliminated the painful waterfall approach to website creation where one team provides final designs to a separate team for implementation.
This shows everyone having an equal seat at the same table. UX, design, and front and back-end development are equally important players. We employ a robust internal peer review process where information architecture, wireframes and design alike are all vetted internally before they are sent as deliverables for clients.
The design process itself starts at literally the smallest level, which takes me to the second pillar...

It Begins With Atoms

Atomic Design is a way of thinking about design through the metaphor of chemistry. Atoms equal small design building block s— a form field, a headline style — that combine to create entire website templates.
Atomic design is a useful construct when approaching responsive design. It also aligns with how developers think about building a site. Let's look at an example.
Start with the smallest building blocks - thumbnail image, headline, byline, date, excerpt.
Build these into a module to be used on the site, like a featured lead story module you might see on the home page.
Continue to build sets of modules to be used throughout the site.
Create templates with sets of your modules.
Build a website from a series of templates.

Benefits

  • Ensures consistency with colors, sizes, and typography
  • Creates a clear plan for building the site
  • Promotes simpler, reusable code
  • It's how developers think
Simplifies CSS, which reduces complexity of maintenance and page size/load time. Developers are involved from the start and see a clear path to building the site. Fewer "snowflakes". If using something like WordPress, easy to see how to create template includes to be reused.
It doesn't stifle creativity. It focuses it on the type of product that you're trying to build, which has a clear goal of delivering information. It creates focus for both design and development on that common goal.
This approach is very object-oriented, which developers understand. This takes us to the third pillar, which is how we actually build these designs. This is defined by one phrase...

The Browser is Best

With design and development under one roof, we can work differently from the traditional approach to web design where a Photoshop file is produced and handed off to developers to slice. This means...

No Photoshop

(OK, very little Photoshop)

While we are likely to produce some key designs in Photoshop to illustrate our ideas and to assist conversations with stakeholders, we will work primarily with wireframes and illustrative style tiles to set forth site colors, typography, iconography and other site design elements. The goal is to produce a style guide that can be used to guide implementation.
This process eliminates the “lost in translation” effect that so often plagues the process of moving from Photoshop design to a site’s live design. Our goal is that you see your site coming to life far earlier than in a traditional design process. We recognize this isn't possible for everyone's workflow especially based on the available skill so...

If You Do Photoshop...

  • Don't rasterize or flatten your layers
  • Avoid altering fonts
  • Label your layers
  • Create layers for different states
  • Create a master style PSD
  • Document!
I recognize this isn't always possible. If Photoshop is still required for you, follow these rules to give the developer as clear a path as possible.
- It's better to "Convert to Smart Object", which allows the developer access the core information, like a font's size, weight, and leading.
- Avoid changing a font's kerning, tracking, scale, and shift, and never use alternate glyphs. These are things that are hard or impossible to build and usually unnecessary to effective information presenation.
- Create layers for different states like hover, active, etc.
- Master styles with:
* 6 levels of headings
* links (include hover state, too)
* basic paragraph (mind the leading/line-height as well as spacing before and after the paragraph)
* quotation
* bulleted lists
* numbered lists
* tables
- Include a document with notes on anything not-so-obvious, like "heading font is Open Sans Condensed Bold, 30pt, 40pt leading, color #111111" or "rounded corners are all a 5px radius" - But regardless of how you have constructed the design, there is one other key place where designers often fail developers and that is remembering our fourth pillar...

Content is King

Every decision in building a website relates to delivering content such that it is more findable, appealing and loads faster.
But what designers often forget to do is use...

Real Content

Designers love to use content that lines up perfectly with their designs. Never in the history of a news site has that ever worked out. Take real content from the site and use it.
A common example of how this fails is...

Overlays

Overlays very often fail unless they're used for something with a predictably short length. When we implemented this design, they were all nice two line headlines. That never works in practice.
In the second example, you can maybe see 15% of the image on the left and Harvey Weinstein is just barely peeking over the image on the right. In these cases, the overlay might actually be fortuitous, but regardless you're seeing less than a quarter of these images.
We actually spent weeks tweaking a complex Javascript algorithm that set the font size based on character count. It inevitably failed because variations on word length and spacing would throw it off. This is about to all be scrapped for a new design that simply puts the text below the image.
And speaking of text, it's also important to be realistic about...

Fonts

  • Make sure they exist
  • Don't overuse custom fonts
  • Use them all from the same service
  • Another reason to design in-browser
- Don't commit to something if it doesn't exist online. You can get it converted, but it can be a process involving licensing fees.
- Developers will get blamed when the site loads slowly but they can't really help it if you're loading 1MB of fonts. It's easy to take advantage of all the fonts in TypeKit, but keep it simple. One or two distinct faces are enough to distinguish you visually.
- Make sure they're all on the same service (like TypeKit or Google Web Fonts) so you're only loading them once and make sure they're all available and licensed.
- Fonts in comps never look exactly the same as on the web
Altogether, the important point to remember is...

Design for Failure

I'm sure many of you have heard this before. It's been an engineering practice for decades where failure is not an option, such as in building bridges and tunnels. It's generally how the (good) hosting services we rely upon are constructed. It goes for visual design as well.
When you're thinking about content, think about those common failure scenarios and make sure that your designs degrade smoothly if the wrong text length or the wrong size image is used. It means easier code to maintain because developers aren't coding for exceptions. That's where code becomes complex and prone to bugs. These bugs have to get handled by developers and they're avoidable if you think about how the design will fail from the start.
And the design will fail in more ways than even now if you don't remember our fifth pillar which is...

Think Responsive

Today, designs are responsive by rule, and are device and browser agnostic. We can’t assume users need the same content on their phone as they do on their desktop. Developers need to work within a framework to make responsive designs feasible to build and maintain, so from the very start...

Design on a Grid

Understand how the grid works and what the natural flow of elements are between breakpoints. Columns tend to collapse and then stack. Showing and hiding elements is easy. Moving to an entirely different area of the page or using a completely different design for different devices is harder. It means writing more code and making your page heavier. Think about reuse and simplicity.

Design to ∞

Design to infinity, and include repeatable patterns when necessary. Let's say your photoshop canvas is 1200px wide, and you have an 1100px-wide website, with a graphical background behind it. What will that background look like if the user has a 27" monitor and they can see over 2,500 pixels of width?
Should elements located in the corners like the logo and navigation keep extending out forever or should they stop at a reasonable width so users on a high res monitor don't have to drag their mouse all the way out to the corner to use them?
Think about these decisions so developers don't have to wonder about them later.

Design all states

There is nothing worse than getting a design and realizing a menu is missing, or a modal window, or something that could equally affect the amount of time required to build a feature. Design interactivity like hover and active states. What should happen in the navigation when an item is hovered-over? What should the sub-menus look like? What should a basic link look like on hover?
What elements are visible at each breakpoint on the grid? Do elements flow logically as they collapse? Make sure these make sense and that the developers have a clear plan to build this within the grid you've chosen. Even if you've remembered absolutely everything, inevitably something will change so remember our last pillar which is...

Launch is not the Finish Line

The design will continue to evolve. New features will be built. You need to establish rules to guide developers through that process.

Create Guidelines

  • Style Tiles
  • Style Guide
Scenarios will exist that weren't explicitly covered in the designs, even during the initial implementation. These will only become more frequent as ongoing site maintenance takes place and new features are added. Don't make developers guess, give them a guide.
Style Tiles are a design deliverable consisting of fonts, colors and interface elements that communicate the essence of a visual brand for the web. They help form a common visual language between the designers and developers (as well as the stakeholders/clients). Style Tiles are similar to the paint chips and fabric swatches an interior designer gets approval on before designing a room. (http://styletil.es/)
Style Guides cover everything else - fonts, colors, spacing, common elements, etc. This is the definitive reference to be referred to when making any implementation decision.

Conclusions

Part of what makes our design process successful is that everyone understands code to some degree. This is important because...

Benefits designers and developers

This makes sites easier to build and maintain for developers, but also ensures that your designs will be better executed. If the design is sensible and achievable, developers won't need to compromise.

Don't design the impossible

There is nothing worse as a developer than spending multiple cycles tweaking a purely visual element. A news site needs to present information effectively first and foremost. Think about how to be practical as much as beautiful.

Learn some code

Everyone on our team knows a little code. It helps them to understand what's possible and what isn't. If you learn a little code, it will be like getting a new version of photoshop. All of a sudden, you'll find a hundred new tools they can work with and that you can design more interactively.

Work Together

The best way to design FOR development
is to design WITH developers

The best way to design FOR development is to design WITH developers. Work together and you'll build better products faster.
